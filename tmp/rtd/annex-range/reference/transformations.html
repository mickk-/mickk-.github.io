Transformations¶

An individual range can be transformed to give rise to a new range. For instance, auto reversed = reverse(rng); produces a range reversed which elements are in reverse order of that of rng.

StorableTypes{Functor, ...Args}
constexpr defer_range<Functor, Args...> operator()(Functor &&functor, Args&&... args) const¶
Notation
auto&& rng = invoke(std::forward<Functor>(functor), std::forward<Args>(args)...);

Defer a range-producing computation. This makes it possible to create a model of Range that serves as a stand-in for the future result rng of invoking functor on args…. Consider the following:

// (1)
auto filtered_rng = filter(pred, input);
if(...some condition...) {
    // use filtered_rng
} else {
    // ignore filtered_rng
}

This fragment may not behave as the programmer intended: if input is non-finite and it contains no element satisfying pred, then the call to filter at (1) will not terminate. This would happen even if filtered_rng were not needed by the rest of the program, e.g. no element access were peformed.

While the example could perhaps be fixed by only constructing filtered_rng when it is known it will be needed (and presumably in such situations construction will be guaranteed to terminate), this is not always an option to the programmer. defer, as its name suggests, delays range construction to the latest point possible, i.e. when an actual range operation is performed on context and positions.

In other words, defer can be thought of as the tool to transform a strict range-producing function into a lax range (see Strict and lax ranges) by deferring all additional call complexity (see Call complexity).

Parameters

functor, args... – function and arguments producing a Range, i.e. modelling OnceInvokable<Functor, Args...>

Models

Lax Range with identical properties as that of rng.

Additional construction complexity

None.

Simple context members

defer_context::deferred_functor: functor

defer_context::deferred_args: std::tuple<Args...> { std::forward<Args>(args)... }

map¶

Provided by annex/range/transformation/map.hpp.

Variables{PeekFunctor, Functor}
class peek_functor_overload_type¶

An overload of two functors. Useful for merging two different functors into one, e.g. for map.

The template parameter names are somewhat fanciful as it is ultimately overload resolution that selects which of PeekFunctor or Functor will be used when dispatching an operator() call, not whether the overload is being used for peeking or not.

Requirements

PeekFunctor and Functor must be non-final class types providing at least one operator() overload.

template<StorableType PeekFunctor, StorableType Functor, MoveConstructible Rng>
constexpr map_range<peek_functor_overload_type<std::decay_t<PeekFunctor>, std::decay_t<Functor>>, context_t<Rng>> operator()(PeekFunctor &&peek_functor, Functor &&functor, Rng rng) const¶

Sometimes the difference between peek_element_t<Ctx> and element_t<Ctx> will prevent a given functor from accepting both as arguments. This overload allows passing two mapping functors with the intention that one will be used for peeking and the other for regular accesses.

Note however that what actually happens is that the two functors are merged into one overload (see peek_functor_overload_type). Which functor will be used for peeking and which will be used for accessing will be selected by overload resolution, hence the order of those two arguments is irrelevant.

Equivalent to operator()(std::move(overload), std::forward<Rng>(rng)), where the overload argument is:

auto overload = make_peek_functor_overload_type(std::forward<PeekFunctor>(peek_functor),
                                                std::forward<Functor>(functor) );

Stride over every steps element of rng:

auto evens  = stride(2, ints());
auto sevens = stride(7, ints());

(See ints.)

In the above, evens is a range over the even integers (i.e. \(\left\{ 0, 2, 4, \dots \right\}\)) whereas sevens ranges over the multiples of seven (i.e. \(\left\{ 0, 7, 14, \dots \right\}\)).

See also

incr_n, decr_n

as the result of stride is rng where incr (resp. decr) has been substituted by incr_n (resp. decr_n), up to steps at a time

This constant is a Range function with one or more associated simple models:

template<Variable Proj, Variable Equiv, MultipassContext Ctx> group_range

template<ForwardableType Proj, ForwardableType Equiv, MoveConstructible Rng>
constexpr group_range<Proj, Equiv, context_t<Rng>> operator()(Proj &&proj, Equiv &&equiv, Rng rng) const¶
Notation
using Ctx    = context_t<Rng>;
using peek_t = peek_element_t<Ctx>;
using proj_t = result<meta::as_const<Proj&>, peek_t>;

auto&& ctx = rng.context;

Create a range over the successive groupings of rng. A grouping is a subrange of consecutive elements with a common property or aspect as determined by proj and equiv, e.g.:

auto classes = group([](auto i) { return i / 3; }, std::equal_to<> {}, ints());

(See ints.)

In this example the first element of classes is the grouping \(\left\{ 0, 1, 2 \right\}\) because all three elements are of the form \(0 \times 3 + k \left(0 \le k \lt 3\right)\). It is followed by the grouping \(\left\{ 3, 4, 5 \right\}\) which elements are of the form \(1 \times 3 + k\), and so on.

If the elements of rng are sorted with respect to proj according to an ordering compatible with equiv, then the result of group is a range over the equivalence classes induced by proj and equiv.

Equivalents

Loops

 
auto it   = begin(xs);
auto last = end(xs);
auto grouping = it;
for(; it != last;) {

    auto const& current = proj(*it);

    // start of grouping
    for(; it != last && equiv(current, proj(*it)); ++it) {
        use(*it);
    }
    // end of grouping
}

D

 
// only sorted ranges may be grouped
// sorting requires an order (here, rel instead of equiv)
// and not just an equivalence
sort!((lhs, rhs) => rel(proj(lhs), proj(rhs))(xs).groupBy()

Rust

 
use itertools::Itertools;
// no direct way of specifying equiv
xs.group_by(proj)

Haskell

 
import Data.List (groupBy)
import Data.Function (on)
-- on g f = \x y -> g (f x) (f y)
groupBy (equiv `on` proj) xs

Python

 
# functools.cmp_to_key can be used when a custom comparison is wanted
itertools.groupby(xs, key=proj)

C#

 
// no direct way of specifying equiv
xs.GroupBy(proj)
Parameters

  • proj – A projection to select which property or aspect of the rng elements will be compared for equivalence. Consequently, must satisfy Invokable<meta::as_const<Proj&>, peek_t>.

  • equiv – An equivalence relation over some property or aspect of the rng elements. Consequently, must satisfy Equivalence<meta::as_const<Equiv&>, meta::as_const<proj_t&>, proj_t>.

  • rng – Saveable Range with a multi-pass context. Additionally, proj_t must be a reference type or model Movable.

template<Traversal Trav, typename Element, typename PeekElement = Element, typename Diff = std::ptrdiff_t, bool Saveable = false>
using any_range = bounded_context<any_context<Trav, Element, PeekElement, Diff, Saveable>>¶

A type-erasing container for a Range which itself behaves as one. The template parameters dictate which Range properties the container will expose, and in turn will require of its underlying range.

Template Parameters

  • Trav – the exposed traversal of the underlying range

  • Element – the exposed element type of the underlying range

  • PeekElement – the exposed element type for peeking of the underlying range

  • Diff – the exposed difference type, if any, of the underlying range

  • Saveable – true iff the save capability of the underlying range is to be exposed

There is one convenience alias per traversal:
